
## Project Context — API Discovery & Security Evaluation Platform

### What This Platform Does

This is a custom-built Python platform that discovers all APIs in a target application, classifies them, runs security checks, and generates reports. 
The platform runs as a 9-phase pipeline triggered by:
```powershell
python main.py --domain http://10.20.40.14:7085 --mode passive
```

### Purpose of Each Key File

**`store/schema.py`**
Defines the `APIEntry` dataclass — the central data object for every discovered endpoint. Fields include: endpoint, method, classification, auth_type, data_sensitivity, exposure, environment, owasp_flags, cve_findings, risk_score, inferred_owner, functional_module, api_version, functional_type, tech_stack, parameters, upstream_services, downstream_dependencies, evidence, tags.

**`store/store.py`**
In-memory store. Key method is `upsert(endpoint, method, source, **kwargs)` — if an endpoint already exists it merges the new discovery source and updates fields. Also holds: `store.secrets_found` (list of secret dicts), `store.package_dependencies` (list of parsed package dicts), `store.outbound_api_inventory` (list of outbound API dicts), `store.tech_stack` (dict).

**`pipeline/p01_external_scan/scanner.py`**
External scanner. Uses `aiohttp` for async HTTP, `requests` for baseline probing, `BeautifulSoup` for HTML parsing, `playwright` for headless Chromium. Establishes false-positive baseline first (detects wildcard SPA servers). Runs brute-force against 469,520 path wordlist. Extracts APIs from JS files via regex. Intercepts live network requests via Playwright. Probes well-known paths, GraphQL endpoints, OpenAPI spec paths, and subdomains.

**`pipeline/p02_source_scan/scanner.py`**
Most important file — produces the majority of discovered endpoints. Does 5 things:
1. **Two-pass router mount pre-scan** — finds `app.use('/base', router)` patterns across all JS files to build a `_base_path_map` for prefix resolution
2. **Package manifest parsing** — parses `package.json`, `requirements.txt`, `pom.xml`, `go.mod` for dependency versions → stored in `store.package_dependencies`
3. **Route extraction (inbound APIs)** — verb-aware regex extraction from Express routes, axios calls, fetch calls → 503 endpoints
4. **Outbound API discovery** — captures all `axios.get('https://...')`, `fetch('https://...')` etc calls → classifies each as known integration or generic external/internal → 24 outbound entries stored in `store.outbound_api_inventory`
5. **Secret scanning** — 7 regex patterns for api_key, password, token, bearer, aws_key, private_key → 46 findings stored in `store.secrets_found`

Key design: secrets and outbound calls are stored as raw dicts in store attributes, NOT as APIEntry objects, so they never contaminate the inbound API count.

**`pipeline/p06_classifier/classifier.py`**
Loads governance baseline from `inputs/openapi_specs/`, `inputs/gateway_exports/`, `inputs/baseline.json`. Classifies each endpoint:
- In spec OR gateway OR baseline → Valid
- Appeared after `new_api_since` date → New
- Matches dangerous patterns (/debug, /.env, /actuator) AND not authorized → Rogue
- Everything else → Shadow

**`pipeline/p08_enrichment/enricher.py`**
Enriches all 503 endpoints with: auth type detection, data sensitivity (CRITICAL/HIGH/MEDIUM/LOW keyword matching), exposure (internal/external/partner), environment, functional type, inferred owner via `MODULE_OWNER_MAP`, CVE correlation using version-aware `PACKAGE_CVE_DATABASE` (15 packages with semver ranges), inferred OWASP flags for all 10 categories, and additive risk score 0–100.

Key fix already applied: `e.inferred_owner` is set here from `MODULE_OWNER_MAP` and must be read in the reporter — never overwrite it with a hardcoded string.

**`pipeline/p09_reporter/reporter.py`**
Generates all 6 output files. Key design decisions already implemented:
- Secrets read from `store.secrets_found` directly — NOT filtered from APIEntry list
- `e.inferred_owner` used everywhere — not hardcoded
- `store.outbound_api_inventory` read for the outbound section
- `_build_inbound_outbound_summary()` builds the inbound vs outbound classification section
- Produces `outbound_api_inventory.json` as a dedicated output file
- PDF and Word both have Section 8: Inbound vs Outbound API Classification

### Key Design Decisions Already Made (Do Not Revert)

1. **Secrets are raw dicts in `store.secrets_found`** — never APIEntry objects. Reporter reads them via `getattr(self.store, 'secrets_found', [])`.
2. **Outbound APIs are in `store.outbound_api_inventory`** — separate from inbound. Reporter reads via `getattr(self.store, 'outbound_api_inventory', [])`.
3. **`e.inferred_owner`** is set by the enricher from `MODULE_OWNER_MAP` — reporter must use `e.inferred_owner`, never a hardcoded fallback string like `"Pending ownership review"`.
4. **Deduplication in outbound** is by `host + path_prefix` — not just host. This allows distinct APIs on the same server to appear as separate entries.
5. **Noise filtering** — method `N/A`, file-system paths, ML model files, malformed URLs are filtered from the inbound inventory in both enricher and reporter.
6. **CVE correlation is version-aware** — uses parsed semver tuples from `store.package_dependencies`, only fires if `installed_version < fixed_version`. Falls back to framework-level pattern matching only if no package manifest was found.
7. **Route extraction is position-independent** — groups are scanned for HTTP verb and path independently, not by position. `fetch()` with no method defaults to GET.
